## 1. 文档概述

### 1.1 文档目的

本文档定义 APP 自动化测试平台的业务需求、技术架构、模块设计、接口规范、数据模型、部署方案、代码规范及异常处理机制，为开发、测试、运维团队提供统一的执行标准。

### 1.2 核心目标

- 支持用户通过自然语言指令驱动 APP 自动化测试
- 实现测试脚本自动生成、智能纠错、状态自愈
- 降低自动化测试门槛，无需手动编写 Maestro 脚本
- 适配 APP 页面元素变化，提升测试稳定性
- 生成专业、可落地的测试报告

### 1.3 适用范围

| 角色 | 职责 |
|---|---|
| 开发团队 | 平台开发、模块集成、接口对接 |

## 2. 需求定义

### 2.1 功能需求

#### FR-001 自然语言指令输入

- 用户通过 Web 端输入自然语言测试指令
- 支持中文、英文指令
- 支持指令模板（快捷填充常见测试场景）
- 输入校验：非空、长度限制（≤500 字符）、敏感词过滤

#### FR-002 意图理解与解析

- 识别测试场景类型（登录、搜索、下单、注册等）
- 提取槽位信息（APP 名称、操作目标、输入内容、预期结果）
- 将自然语言转换为结构化指令
- 支持多轮对话补全缺失信息（如用户未提供密码时主动询问）

#### FR-003 脚本匹配与复用

- 基于结构化指令在脚本库中进行语义匹配
- 匹配算法：向量相似度（Embedding + Cosine Similarity），阈值 ≥ 0.85
- 命中缓存脚本时直接复用，未命中时触发脚本生成
- 支持脚本版本管理（同一场景可存在多个版本）

#### FR-004 脚本自动生成

- LLM 生成伪代码步骤 → 模板引擎转换为 Maestro YAML 脚本
- 脚本预校验（语法检查、元素格式合法性）
- 生成失败时自动重试（最多 2 次）
- 生成的脚本自动入库（Redis 缓存 + MySQL 持久化）

#### FR-005 脚本执行与调度

- 支持单设备执行、多设备并行执行
- 执行前自动初始化环境（设备连接、APP 安装、日志开启）
- 线性执行脚本步骤，每步执行前后触发视觉校验
- 实时推送执行状态（WebSocket）

#### FR-006 视觉校验

- 每步执行前后截图并分析
- 识别页面元素（ID、文本、坐标、类型）及页面状态（加载中/弹窗/正常）
- 与预期状态对比，输出校验结果
- 降级方案：当视觉模型不可用时，回退到 Accessibility Tree 解析

#### FR-007 智能纠错与自愈

- 元素未找到 → LLM 重新定位元素，更新脚本
- 页面跳转异常 → 回溯上一步重新执行
- 弹窗干扰 → 自动插入弹窗关闭步骤
- 加载超时 → 延长等待时间（最多 2 倍）
- 输入失败 → 清空后重新输入
- 不可修复错误（APP 崩溃）→ 终止执行，生成报告
- 同一步骤最多重试 2 次

#### FR-008 测试报告生成

- 自动生成 HTML 格式报告（支持导出 PDF）
- 报告内容：错误概览 → 详细信息 → 截图对比 → 修复建议 → 可执行修复脚本
- 报告存储并生成唯一访问链接
- 支持 Web 端通知 + 邮件推送

#### FR-009 脚本管理

- 脚本列表查看、搜索、筛选
- 脚本详情查看与手动编辑
- 脚本删除（软删除）
- 脚本执行历史查看

#### FR-010 设备管理

- 已连接设备列表展示
- 设备状态监控（在线/离线/执行中）
- 设备分配与释放
- 支持 Android 设备（iOS 后续扩展）

### 2.2 非功能需求

| 编号 | 类别 | 要求 |
|---|---|---|
| NFR-001 | 性能 | 指令响应时间 ≤ 3s（意图理解 + 脚本匹配） |
| NFR-002 | 性能 | 脚本生成时间 ≤ 10s（LLM 生成 + 模板转换 + 校验） |
| NFR-003 | 性能 | 单步执行时间 ≤ 步骤预期等待时间（默认 ≤ 10s） |
| NFR-004 | 并发 | 同时支持 ≥ 50 台设备执行任务，支持横向扩展 |
| NFR-005 | 可用性 | 系统可用性 ≥ 99.5%（月度） |
| NFR-006 | 安全 | 用户数据 AES-256 加密存储，接口签名验证 |
| NFR-007 | 安全 | 截图自动脱敏（手机号、密码、身份证号） |
| NFR-008 | 可维护性 | 核心模块单元测试覆盖率 ≥ 80% |
| NFR-009 | 兼容性 | 支持 Android 8.0+ 设备 |
| NFR-010 | 可观测性 | 全链路日志追踪，支持按 taskId 检索 |

### 3.3 LLM 抽象适配层设计

为避免与特定模型耦合，引入统一的 LLM 适配层：

```python
# app/llm/base.py
from abc import ABC, abstractmethod
from pydantic import BaseModel

class LLMResponse(BaseModel):
    content: str
    model: str
    usage: dict  # token 用量

class BaseLLMProvider(ABC):
    """LLM 提供者抽象基类"""

    @abstractmethod
    async def chat(self, messages: list[dict], **kwargs) -> LLMResponse:
        ...

    @abstractmethod
    async def health_check(self) -> bool:
        ...
```

```python
# app/llm/factory.py
class LLMFactory:
    """LLM 工厂，支持主备切换"""

    _providers: dict[str, type[BaseLLMProvider]] = {}

    @classmethod
    def register(cls, name: str, provider: type[BaseLLMProvider]):
        cls._providers[name] = provider

    @classmethod
    def create(cls, name: str, **kwargs) -> BaseLLMProvider:
        if name not in cls._providers:
            raise ValueError(f"Unknown LLM provider: {name}")
        return cls._providers[name](**kwargs)
```

配置文件中指定主备模型：

```yaml
# config/llm.yaml
llm:
  primary: "openai-gpt4o"
  fallback: "anthropic-claude"
  timeout: 30
  max_retries: 2
  providers:
    openai-gpt4o:
      api_key: "${OPENAI_API_KEY}"
      model: "gpt-4o"
      max_tokens: 4096
    anthropic-claude:
      api_key: "${ANTHROPIC_API_KEY}"
      model: "claude-sonnet-4-20250514"
      max_tokens: 4096
```

## 4. 核心流程设计

### 4.1 主执行流程

```
用户输入指令
     ↓
[步骤1] 指令预处理（校验、日志记录）
     ↓
[步骤2] 意图理解（LLM 解析 → 结构化指令）
     ↓
[步骤3] 脚本匹配（Redis → MySQL → 未命中则生成）
     ↓
[步骤4] 脚本生成（LLM 伪代码 → 模板引擎 → Maestro YAML → 预校验）
     ↓
[步骤5] 脚本执行（Maestro 驱动设备，逐步执行）
     ↓ 每步执行前后
[步骤6] 视觉校验（截图 → AutoGLM 分析 → 状态对比）
     ↓ 校验失败
[步骤7] 智能纠错（LLM 分析 → 最小化修复 → 重试，最多2次）
     ↓
[步骤8] 结果反馈 + 报告生成
```

### 4.2 步骤详细说明

#### 步骤 1：指令预处理

```python
# 伪代码
def preprocess_instruction(user_id: str, instruction: str) -> str:
    # 1. 非空校验
    if not instruction.strip():
        raise ValidationError("指令不能为空")
    # 2. 长度校验
    if len(instruction) > 500:
        raise ValidationError("指令长度不能超过500字符")
    # 3. 敏感词过滤（白名单机制）
    if contains_dangerous_keywords(instruction):
        raise SecurityError("指令包含危险操作")
    # 4. 记录操作日志
    log_operation(user_id, instruction, timestamp=now())
    return instruction.strip()
```

#### 步骤 2：意图理解

LLM 输入 Prompt 模板：

```
你是一个 APP 自动化测试专家。请分析以下用户指令，输出 JSON 格式的结构化指令。

用户指令："{instruction}"

输出格式：
{
  "intent": "login|search|purchase|register|...",
  "app_package": "包名（如能推断）",
  "steps": [
    {
      "action": "launch|tap|input|swipe|wait|assert",
      "target": "元素描述",
      "value": "输入值（如有）",
      "expected": "预期结果（如有）"
    }
  ],
  "missing_info": ["缺失的必要信息"]
}
```

#### 步骤 3-4：脚本匹配与生成

匹配流程：
1. 将结构化指令通过 Embedding 模型转为向量
2. 在 Redis 中查询向量缓存（TTL 24h），命中则返回
3. 未命中则查询 MySQL 脚本库（Cosine Similarity ≥ 0.85）
4. 仍未命中则触发 LLM 生成伪代码 → 模板引擎转换

Maestro 脚本模板示例：

```yaml
# templates/maestro_base.yaml
appId: ${app_package}
---
- launchApp
- assertVisible:
    id: "${initial_screen_element}"
    timeout: 10000
# 以下步骤由模板引擎动态填充
${generated_steps}
```

#### 步骤 5-6：执行与视觉校验

每步执行的生命周期：

```
执行前截图 → AutoGLM 分析当前页面 → 前置校验
     ↓ 通过
执行 Maestro 步骤
     ↓
执行后截图 → AutoGLM 分析结果页面 → 后置校验
     ↓ 通过          ↓ 失败
  下一步骤        触发智能纠错
```

#### 步骤 7：智能纠错

| 异常类型 | 检测方式 | 修复策略 | 最大重试 |
|---|---|---|---|
| 元素未找到 | AutoGLM 识别不到目标元素 | LLM 重新定位（文本/坐标匹配），更新脚本 | 2 次 |
| 页面跳转异常 | 预期页面与实际不匹配 | 回溯上一步，重新执行路径 | 1 次 |
| 弹窗干扰 | AutoGLM 识别到非预期弹窗 | 插入弹窗关闭步骤，再执行原步骤 | 1 次 |
| 加载超时 | 超过等待时间未加载完成 | 延长等待时间（最多 2 倍） | 1 次 |
| 输入失败 | 输入后元素内容未更新 | 清空后重新输入，检查输入法状态 | 1 次 |
| APP 崩溃 | 设备返回崩溃日志 / 黑屏 | 终止执行，生成错误报告 | 0 次 |

纠错修复原则：最小化修复，仅修改当前步骤，不重写整个脚本。修复后的脚本存入 Redis（带 `_fixed` 后缀），不覆盖原始脚本。

## 5. 数据模型设计

### 5.1 MySQL 表结构

#### users 表

```sql
CREATE TABLE `users` (
    `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    `username` VARCHAR(64) NOT NULL UNIQUE,
    `password_hash` VARCHAR(255) NOT NULL,
    `role` ENUM('admin', 'tester', 'developer', 'viewer') NOT NULL DEFAULT 'tester',
    `email` VARCHAR(128),
    `status` TINYINT NOT NULL DEFAULT 1 COMMENT '1=启用 0=禁用',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_role` (`role`),
    INDEX `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### scripts 表

```sql
CREATE TABLE `scripts` (
    `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    `script_id` VARCHAR(36) NOT NULL UNIQUE COMMENT 'UUID',
    `intent` VARCHAR(64) NOT NULL COMMENT '测试意图',
    `structured_instruction` JSON NOT NULL COMMENT '结构化指令',
    `instruction_embedding` BLOB COMMENT '指令向量（用于相似度匹配）',
    `pseudo_code` TEXT COMMENT 'LLM 生成的伪代码',
    `maestro_yaml` TEXT NOT NULL COMMENT 'Maestro 脚本内容',
    `version` INT NOT NULL DEFAULT 1,
    `hit_count` INT NOT NULL DEFAULT 0 COMMENT '命中次数',
    `status` ENUM('active', 'deprecated', 'draft') NOT NULL DEFAULT 'active',
    `created_by` BIGINT UNSIGNED NOT NULL,
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_intent` (`intent`),
    INDEX `idx_status` (`status`),
    INDEX `idx_hit_count` (`hit_count` DESC),
    FOREIGN KEY (`created_by`) REFERENCES `users`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### tasks 表

```sql
CREATE TABLE `tasks` (

    `task_id` VARCHAR(36) NOT NULL UNIQUE COMMENT 'UUID',

    `completed_steps` INT DEFAULT 0,

    `report_url` VARCHAR(512) COMMENT '报告链接',
    `started_at` DATETIME,
    `finished_at` DATETIME,
    `duration_ms` INT COMMENT '执行耗时（毫秒）',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX `idx_user_id` (`user_id`),
    INDEX `idx_status` (`status`),
    INDEX `idx_created_at` (`created_at` DESC),
    FOREIGN KEY (`user_id`) REFERENCES `users`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### task_steps 表


#### devices 表

```sql

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
